## 🚀 Процесс внесения вклада

### 1. Найдите задачу

**Для новичков:**
- Ищите задачи с меткой `good-first-issue`
- Ищите задачи с меткой `help-wanted`

**Для опытных:**
- Открытые задачи с меткой `feature`
- Баги с меткой `bug`

**Где искать:**
- [GitHub Issues](https://github.com/fab-orbita/shield/issues)
- [GitHub Projects](https://github.com/fab-orbita/shield/projects)

---

### 2. Форкните репозиторий

```bash
# Форкните репозиторий на GitHub
# Затем клонируйте свой форк
git clone https://github.com/YOUR_USERNAME/shield.git
cd shield

# Добавьте upstream (оригинальный репозиторий)
git remote add upstream https://github.com/fab-orbita/shield.git
3. Создайте ветку
bash
# Для новой функции
git checkout -b feature/название-функции

# Для исправления бага
git checkout -b fix/название-бага

# Для документации
git checkout -b docs/название-изменения
4. Внесите изменения
bash
# Установите зависимости
npm install

# Запустите тесты
npm test

# Запустите сборку
npm run build

# Запустите линтер
npm run lint

# Запустите форматтер
npm run format
5. Напишите тесты
typescript
// Пример теста для нового функционала
describe('New Feature', () => {
    it('should work correctly', () => {
        const result = newFeature()
        expect(result).toBe('expected')
    })
})
6. Обновите документацию
Обновлен README.md

Обновлена API документация

Добавлены примеры

7. Создайте Pull Request
bash
# Добавьте изменения
git add .

# Сделайте коммит (следуйте Conventional Commits)
git commit -m "feat: add new security header"

# Отправьте в ваш форк
git push origin feature/название-функции

# Создайте Pull Request на GitHub
📝 Правила оформления коммитов
Мы используем Conventional Commits:

Типы коммитов:
Тип	Описание
feat:	Новая функция
fix:	Исправление бага
docs:	Документация
style:	Стиль кода
refactor:	Рефакторинг
perf:	Производительность
test:	Тесты
chore:	Сборка, инструменты
Примеры:
bash
feat: add dynamic CSP support
fix: resolve memory leak in AI module
docs: update API reference
refactor: optimize header generation
test: add unit tests for metrics
🔍 Процесс ревью
Что проверяется:
Критерий	Описание
✅ Код	Качество, стиль, читаемость
✅ Тесты	Покрытие, корректность
✅ Документация	Полнота, понятность
✅ Производительность	Отсутствие регрессий
✅ Безопасность	Нет новых уязвимостей

💡 Рекомендации по коду
DO (делайте):
✅ Используйте TypeScript строго
✅ Пишите понятные имена функций
✅ Добавляйте JSDoc комментарии
✅ Следуйте принципам SOLID
✅ Пишите тесты для нового кода
✅ Используйте async/await

DON'T (не делайте):
❌ Не используйте any без необходимости
❌ Не игнорируйте ошибки
❌ Не создавайте большие PR (1-2 задачи)
❌ Не ломайте обратную совместимость без обсуждения
❌ Не оставляйте TODO без объяснения
