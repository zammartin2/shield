## 🚀 Процесс внесения вклада

### 1. Найдите задачу

Для новичков:

- ищите задачи с меткой `good-first-issue`;
- ищите задачи с меткой `help-wanted`;
- начинайте с документации, примеров или небольших исправлений.

Для опытных разработчиков:

- задачи с меткой `feature`;
- баги с меткой `bug`;
- задачи по производительности;
- задачи по безопасности;
- улучшение архитектуры.

Где искать:

- [GitHub Issues](https://github.com/zammartin2/shield/issues)
- [GitHub Pull Requests](https://github.com/zammartin2/shield/pulls)

---

### 2. Форкните репозиторий

Сначала форкните репозиторий на GitHub, затем клонируйте свой форк:

```bash
git clone https://github.com/YOUR_USERNAME/shield.git
cd shield
```

Добавьте оригинальный репозиторий как `upstream`:

```bash
git remote add upstream https://github.com/zammartin2/shield.git
```

Проверьте remotes:

```bash
git remote -v
```

---

### 3. Создайте ветку

Для новой функции:

```bash
git checkout -b feature/название-функции
```

Для исправления бага:

```bash
git checkout -b fix/название-бага
```

Для документации:

```bash
git checkout -b docs/название-изменения
```

Для рефакторинга:

```bash
git checkout -b refactor/название-изменения
```

---

### 4. Установите зависимости

```bash
npm install
```

---

### 5. Внесите изменения

Перед началом работы убедитесь, что проект собирается:

```bash
npm run build
```

Запустите тесты:

```bash
npm test
```

Запустите линтер:

```bash
npm run lint
```

Запустите форматтер:

```bash
npm run format
```

---

### 6. Напишите тесты

Если вы добавляете новую функциональность или исправляете баг, добавьте тесты.

Пример теста:

```typescript
describe('New Feature', () => {
  it('should work correctly', () => {
    const result = newFeature()

    expect(result).toBe('expected')
  })
})
```

---

### 7. Обновите документацию

Если вы добавили новую функцию, изменили API или поведение библиотеки, обновите документацию.

Проверьте:

```markdown
- [ ] Обновлен README.md
- [ ] Обновлена русская документация
- [ ] Обновлена API-документация
- [ ] Добавлены примеры
- [ ] Обновлен CHANGELOG.md
```

---

### 8. Создайте коммит

Добавьте изменения:

```bash
git add .
```

Сделайте коммит по правилам Conventional Commits:

```bash
git commit -m "feat: add new security header"
```

Отправьте ветку в свой форк:

```bash
git push origin feature/название-функции
```

Затем создайте Pull Request на GitHub.

---

## 📝 Правила оформления коммитов

Мы используем **Conventional Commits**.

### Типы коммитов

| Тип | Назначение |
|:---|:---|
| `feat` | Новая функция |
| `fix` | Исправление бага |
| `docs` | Документация |
| `style` | Стиль кода без изменения логики |
| `refactor` | Рефакторинг |
| `perf` | Улучшение производительности |
| `test` | Тесты |
| `chore` | Сборка, инструменты, служебные изменения |
| `ci` | CI/CD |
| `build` | Сборка проекта |
| `security` | Изменения безопасности |

---

### Примеры коммитов

```bash
feat: add dynamic CSP support
fix: resolve memory leak in AI module
docs: update API reference
refactor: optimize header generation
test: add unit tests for metrics
security: improve payload validation
chore: update dependencies
```

## 🔍 Процесс ревью

Каждый Pull Request проходит ревью.

Что проверяется:

| Критерий | Описание |
|:---|:---|
| ✅ Код | Качество, стиль, читаемость |
| ✅ Тесты | Покрытие, корректность |
| ✅ Документация | Полнота, понятность |
| ✅ Производительность | Отсутствие регрессий |
| ✅ Безопасность | Нет новых уязвимостей |
| ✅ Совместимость | Нет случайных breaking changes |

---

## 🎯 Критерии принятия Pull Request

### Обязательно

- Все тесты проходят
- Код соответствует стилю проекта
- Добавлены тесты для нового функционала
- Обновлена документация
- Нет конфликтов с основной веткой
- Соблюдены правила оформления коммитов
- Изменения не ломают публичный API без обсуждения

### Желательно

- Покрытие тестами не ухудшилось
- Производительность не ухудшилась
- Добавлены примеры использования
- Обновлен `CHANGELOG.md`
- PR решает одну конкретную задачу

## 💡 Рекомендации по коду

### ✅ Делайте

- Используйте TypeScript строго
- Пишите понятные имена функций и переменных
- Добавляйте JSDoc-комментарии для публичного API
- Следуйте принципам SOLID
- Пишите тесты для нового кода
- Используйте `async/await`
- Обрабатывайте ошибки явно
- Делайте маленькие и понятные Pull Request
- Проверяйте безопасность изменений

---

### ❌ Не делайте

- Не используйте `any` без необходимости
- Не игнорируйте ошибки
- Не создавайте огромные PR на много разных задач
- Не ломайте обратную совместимость без обсуждения
- Не оставляйте `TODO` без объяснения
- Не добавляйте лишние зависимости без причины
- Не храните секреты в коде
- Не отключайте проверки безопасности ради удобства

### Баги

Используйте такой шаблон:

```markdown

Краткое и понятное описание проблемы.

## Шаги для воспроизведения

1. ...
2. ...
3. ...

## Ожидаемое поведение

Что должно было произойти.

## Фактическое поведение

Что произошло на самом деле.

## Окружение

- OS: Windows / Linux / macOS
- Node.js: версия
- FAB Shield: версия
- Framework: Express / Fastify / Koa / NestJS
```
