## 📋 Введение

На рынке существует несколько подходов к защите Node.js-приложений:

- 🛡️ **Helmet** — популярная библиотека для настройки security-заголовков
- 💰 **Платные WAF** — Cloudflare WAF, AWS WAF, Imperva, Akamai и другие решения
- 🔧 **Самописные решения** — собственные middleware, фильтры и правила
- 📦 **Набор отдельных библиотек** — rate limiter, CSP, headers, logger и другие пакеты по отдельности

**FAB Shield** предлагает другой подход: единый security-фреймворк, который объединяет security-заголовки, CSP, AI-анализ, rate limiting, плагины, метрики и мониторинг.

Этот документ помогает понять, в чем сильные стороны FAB Shield и когда его стоит использовать.

## 🛡️ FAB Shield vs Helmet

### Что делает Helmet

Helmet — популярная библиотека для Node.js, которая помогает настраивать HTTP security-заголовки.

**Сильные стороны Helmet:**

- ✅ простое подключение;
- ✅ хороший набор security-заголовков;
- ✅ бесплатный open-source инструмент;
- ✅ зрелая экосистема;
- ✅ подходит для базового hardening Express-приложений.

---

### Чего не хватает Helmet

Helmet решает важную, но ограниченную задачу — security-заголовки.

В нем нет:

- ❌ AI-защиты;
- ❌ встроенного rate limiting;
- ❌ системы плагинов;
- ❌ метрик безопасности;
- ❌ отчетов;
- ❌ мониторинга угроз;
- ❌ application-level анализа подозрительных запросов.

---

### Подробное сравнение

| Аспект | Helmet | FAB Shield | Преимущество FAB Shield |
|:---|:---:|:---:|:---|
| **Security Headers** | ✅ | ✅ | Более широкий security-layer |
| **CSP** | ✅ | ✅ | Возможность динамического CSP |
| **AI-анализ** | ❌ | ✅ | Дополнительный уровень обнаружения угроз |
| **Rate Limiting** | ❌ | ✅ | Защита от brute force и API abuse |
| **Плагины** | ❌ | ✅ | Расширение под свои задачи |
| **Метрики** | ❌ | ✅ | Видимость security-событий |
| **Отчеты** | ❌ | ✅ | Удобнее для аудита и мониторинга |
| **Модульность** | 🟡 | ✅ | Можно включать только нужные функции |

---

### Вывод

**Helmet** — отличный выбор, если нужны только security-заголовки.

**FAB Shield** подходит, если нужен более широкий security-фреймворк: headers, CSP, AI-анализ, rate limiting, плагины, метрики и мониторинг в одном инструменте.

## 💰 FAB Shield vs платные WAF

### Что такое WAF

**WAF** — Web Application Firewall. Такие решения обычно работают перед приложением и фильтруют подозрительный HTTP-трафик.

Примеры:

- Cloudflare WAF;
- AWS WAF;
- Imperva WAF;
- Akamai WAF;
- другие cloud или enterprise WAF.

---

### Что умеют платные WAF

- ✅ защита от известных атак;
- ✅ DDoS-фильтрация;
- ✅ правила по IP, странам и сигнатурам;
- ✅ мониторинг;
- ✅ отчеты;
- ✅ managed rules;
- ✅ коммерческая поддержка.

---

### Возможные сложности WAF

- ❌ стоимость может быть высокой;
- ❌ настройка может быть сложной;
- ❌ часть решений привязана к cloud-провайдеру;
- ❌ не всегда есть полный контроль над логикой;
- ❌ кастомизация может быть ограниченной тарифом или платформой;
- ❌ WAF не всегда видит внутренний контекст приложения.

---

### Сравнение

| Аспект | Платные WAF | FAB Shield |
|:---|:---:|:---:|
| **Цена** | Часто платно | Бесплатно |
| **Настройка** | Средняя / сложная | Простая |
| **Гибкость** | Зависит от провайдера | Высокая |
| **Cloud-зависимость** | Часто есть | Нет |
| **Application-level контекст** | Ограничен | Есть |
| **AI-анализ** | Зависит от решения | ✅ |
| **Плагины** | Зависит от провайдера | ✅ |
| **Open Source** | Обычно нет | ✅ |
| **Self-hosted** | Не всегда | ✅ |
| **Коммерческая поддержка** | ✅ | Зависит от модели проекта |

---

### Вывод

Платный WAF может быть необходим для крупных enterprise-систем, DDoS-защиты и инфраструктурной фильтрации.

**FAB Shield** не обязан заменять WAF полностью. Его сильная сторона — защита на уровне Node.js-приложения, где доступен контекст маршрутов, request body, headers, плагинов и внутренней логики.

Лучший вариант для критичных систем часто такой:

```text
External WAF
    +
Reverse Proxy
    +
FAB Shield
    +
Secure Application Architecture
```

## 🔧 FAB Shield vs своё решение

### Что нужно сделать при самописной защите

Если писать security-layer самостоятельно, обычно нужно:

- настроить security-заголовки;
- реализовать CSP;
- сделать rate limiting;
- добавить фильтрацию подозрительных payload;
- продумать логирование;
- добавить метрики;
- написать тесты;
- поддерживать документацию;
- следить за новыми угрозами;
- исправлять ошибки;
- регулярно обновлять код.

---

### Основные затраты

| Зона | Что потребуется |
|:---|:---|
| **Разработка** | Время backend-разработчика |
| **Безопасность** | Security expertise |
| **Тесты** | Unit, integration, security tests |
| **Документация** | Описание API и правил |
| **Поддержка** | Исправления и обновления |
| **Мониторинг** | Логи, метрики, alerts |
| **Аудит** | Проверка корректности реализации |

---

### Почему FAB Shield может быть лучше

| Аспект | Своё решение | FAB Shield |
|:---|:---:|:---:|
| **Время внедрения** | Долго | Быстро |
| **Стоимость** | Зависит от команды | Бесплатно |
| **Качество** | Зависит от реализации | Единый open-source инструмент |
| **Документация** | Нужно писать | Уже есть |
| **Поддержка** | Внутренняя команда | Сообщество |
| **Обновления** | Нужно делать вручную | Развиваются в проекте |
| **Риски ошибок** | Выше | Ниже при правильной настройке |

---

### Вывод

Самописное решение может быть оправдано, если у компании есть сильная security-команда и уникальные требования.

Во многих случаях быстрее и безопаснее начать с готового фреймворка, а специфичные требования закрывать через плагины и конфигурацию.

## 🧠 Как выбрать решение

### Используйте Helmet, если:

- нужны только security-заголовки;
- проект простой;
- не нужны метрики, AI-анализ и плагины;
- вы хотите минимальную зависимость.

---

### Используйте FAB Shield, если:

- нужен единый security-layer;
- нужны security-заголовки + rate limiting;
- нужна расширяемость через плагины;
- нужен мониторинг security-событий;
- нужен AI-анализ подозрительных запросов;
- проект на Express, Fastify, Koa или NestJS;
- нужно быстро усилить безопасность Node.js-приложения.

---

### Используйте WAF, если:

- нужна защита на периметре;
- нужна DDoS-защита;
- нужна фильтрация до попадания трафика в приложение;
- есть enterprise-требования;
- нужен managed security-service;
- важна коммерческая поддержка.

---

### Пишите своё решение, если:

- есть уникальные требования;
- есть опытная security-команда;
- нужно полное управление каждым правилом;
- готова долгосрочная поддержка;
- есть ресурсы на тестирование и аудит.
