# 🔄 Потоки данных в FAB Shield

> Подробное описание data flow внутри **FAB Shield**  
> Современного security-фреймворка для Node.js-приложений.

**Дата:** 2026-07-01  
**Автор:** Фабрициус Владимир Николаевич

## 📋 Введение

Понимание потоков данных в **FAB Shield** — ключ к правильной настройке, отладке и эффективному использованию фреймворка.

В этом документе описано:

- как запрос проходит через FAB Shield;
- как работает AI-анализ;
- как собираются метрики;
- как выполняются плагины;
- как происходит обнаружение и обработка угроз;
- как формируются логи, отчеты и уведомления.

| **Headers** | Добавляются security-заголовки |
| **CSP** | Генерируется Content Security Policy |
| **AI Analysis** | Выполняется анализ подозрительных признаков |
| **Threat Detection** | Рассчитывается риск и выбирается действие |
| **Application** | Запрос передается в приложение |
| **Response** | Ответ возвращается клиенту |
| **Metrics** | Событие записывается в метрики и логи |

### Что анализируется

| Источник данных | Примеры |
|:---|:---|
| **URL** | path, query params, suspicious patterns |
| **Headers** | User-Agent, Authorization, Content-Type |

| **IP** | reputation, frequency, geography |
| **Behavior** | request rate, sequence, route usage |
| **Session** | repeated failures, suspicious state |

```text
Metrics updated
```

## ✅ Рекомендации

- Подключайте FAB Shield до основных маршрутов.
- Включайте `reportOnly` для CSP перед жесткой блокировкой.
- Не логируйте чувствительные данные.
- Для production включайте мониторинг и алерты.
- Проверяйте метрики после включения новых правил.
- Используйте rate limiting для публичных API.
- Плагины должны быть быстрыми и не блокировать event loop.
- Для нескольких инстансов используйте shared storage для rate limiting и метрик.
