# 🧩 Модули FAB Shield

> Архитектура и описание модулей **FAB Shield**  
> Современного security-фреймворка для Node.js-приложений.

---

**Версия:** 1.0.0  
**Дата:** 2026-07-01  
**Автор:** Фабрициус Владимир Николаевич  
**Компания:** ООО «Деворбит» (DEVORBIT LLC)

---

## 📋 Введение

**FAB Shield** построен по модульному принципу.

Это позволяет:

- 🔧 использовать только нужные компоненты;
- 🚀 легко расширять функциональность;
- 🔄 независимо развивать отдельные модули;
- 🧪 упрощать тестирование;
- 📦 подключать дополнительные возможности постепенно;
- 🛡️ строить гибкий security-layer под конкретный проект.

## 📦 Описание модулей

## 1. 🧠 Core Module

**Назначение:** базовый модуль, управляющий всем фреймворком.

### Ответственность

- инициализация фреймворка;
- управление конфигурацией;
- координация других модулей;
- обработка жизненного цикла;
- регистрация middleware;
- централизованная обработка ошибок.

### Пример API

```typescript
class Core {
  constructor(config: ShieldConfig)

  init(): Promise<void>

  destroy(): Promise<void>

  getConfig(): ShieldConfig

  updateConfig(config: Partial<ShieldConfig>): void
}
```

---

## 2. 🔒 Headers Module

**Назначение:** управление HTTP security-заголовками.

### Ответственность

- установка security-заголовков;
- настройка HSTS;
- настройка Referrer-Policy;
- настройка Permissions-Policy;
- удаление небезопасных заголовков;
- добавление кастомных заголовков.

### Пример API

```typescript
class HeadersModule {
  apply(req: Request, res: Response): void

  setHeader(name: string, value: string): void

  removeHeader(name: string): void

  getHeaders(): Record<string, string>
}
```

### Поддерживаемые заголовки

| Заголовок | Назначение |
|:---|:---|
| `Content-Security-Policy` | Контроль источников ресурсов |
| `Strict-Transport-Security` | Принудительное использование HTTPS |
| `X-Frame-Options` | Защита от clickjacking |
| `X-Content-Type-Options` | Защита от MIME sniffing |
| `Referrer-Policy` | Контроль передачи referrer |
| `Permissions-Policy` | Ограничение browser APIs |
| `Cross-Origin-Opener-Policy` | Изоляция browsing context |
| `Cross-Origin-Resource-Policy` | Защита cross-origin ресурсов |
| `Cross-Origin-Embedder-Policy` | Изоляция embedded resources |
| `X-DNS-Prefetch-Control` | Контроль DNS prefetch |
| `X-Download-Options` | Защита загрузок |
| `X-Permitted-Cross-Domain-Policies` | Ограничение cross-domain policies |
| `X-Powered-By` | Удаляется для сокрытия технологии |

---

## 3. 🛡️ CSP Module

**Назначение:** управление Content Security Policy.

### Ответственность

- генерация CSP;
- поддержка динамического CSP;
- управление nonce;
- настройка trusted sources;
- обработка CSP report-only режима;
- прием отчетов о нарушениях.

### Пример API

```typescript
class CSPModule {
  generate(req: Request, res: Response): string

  setDirective(name: string, value: string[]): void

  getDirective(name: string): string[]

  enableReporting(uri: string): void
}
```

### Поддерживаемые директивы

| Директива | Назначение |
|:---|:---|
| `default-src` | Источник по умолчанию |
| `script-src` | Источники JavaScript |
| `style-src` | Источники CSS |
| `img-src` | Источники изображений |
| `font-src` | Источники шрифтов |
| `connect-src` | Источники fetch/XHR/WebSocket |
| `frame-src` | Источники iframe |
| `frame-ancestors` | Кто может встраивать страницу |
| `base-uri` | Разрешенные base URI |
| `form-action` | Разрешенные form submit targets |
| `object-src` | Источники object/embed/applet |
| `media-src` | Источники аудио и видео |
| `worker-src` | Источники worker |
| `manifest-src` | Источники manifest |

## 4. 🤖 AI Module

**Назначение:** интеллектуальная защита и анализ подозрительной активности.

### Ответственность

- обнаружение аномалий;
- прогнозирование угроз;
- анализ поведения;
- анализ payload;
- проверка User-Agent;
- автоматическая адаптация правил;
- помощь в выявлении нестандартных атак.

### Пример API

```typescript
class AIModule {
  analyze(req: Request): Promise<AnalysisResult>

  detectAnomaly(data: unknown): Promise<AnomalyResult>

  predictThreat(data: unknown): Promise<ThreatPrediction>

  adapt(config: ShieldConfig): Promise<void>
}
```

### Подмодули

| Подмодуль | Назначение |
|:---|:---|
| **XSS Detector** | Поиск XSS-паттернов |
| **SQL Injection Detector** | Поиск SQL Injection-паттернов |
| **User-Agent Analyzer** | Анализ User-Agent |
| **IP Reputation Checker** | Проверка IP-репутации |
| **Behavioral Analyzer** | Анализ поведения запросов |
| **Content Analyzer** | Анализ body, query и headers |

## 5. 🔌 Plugins Module

**Назначение:** управление системой плагинов.

### Ответственность

- регистрация плагинов;
- отключение плагинов;
- управление жизненным циклом;
- выполнение plugin middleware;
- передача конфигурации в плагины;
- изоляция ошибок плагинов.

### Пример API

```typescript
class PluginsModule {
  register(plugin: Plugin): void

  unregister(name: string): void

  getPlugins(): Plugin[]

  execute(req: Request, res: Response): Promise<void>
}
```

### Пример плагина

```typescript
const customPlugin = {
  name: 'custom-security',
  version: '1.0.0',

  middleware: (req, res, next) => {
    res.setHeader('X-Custom-Security', 'enabled')
    next()
  }
}
```

## 8. ⚡ Rate Limiting Module

**Назначение:** ограничение частоты запросов.

### Ответственность

- ограничение запросов;
- настройка лимитов;
- блокировка нарушителей;
- защита от brute force;
- защита API от abuse;
- поддержка разных стратегий лимитов.

### Пример API

```typescript

  check(req: Request): boolean

  increment(key: string): void

  reset(key: string): void

  getLimit(key: string): number
}
```

### Примеры стратегий

| Стратегия | Описание |
|:---|:---|
| IP-based | Лимит по IP |
| User-based | Лимит по user id |
| Route-based | Лимит по маршруту |
| Token-based | Лимит по API token |

| Fixed window | Фиксированное окно |

## 🔄 Взаимодействие модулей

### Общая схема

```text
Request
   ↓
Core
   ↓
Headers
   ↓
CSP
   ↓
AI Analysis
   ↓
Plugins
   ↓
Rate Limiting
   ↓
Rules
   ↓
Metrics
   ↓
Application
   ↓
Response
   ↓
Reporting / Logging / Monitor
```

---

### Поток данных

```text
1. Core получает запрос
2. Headers устанавливает security-заголовки
3. CSP генерирует политику безопасности
4. AI анализирует запрос
5. Plugins выполняют дополнительную логику
6. Rate Limiting проверяет лимиты
7. Rules применяют кастомные правила
8. Metrics собирает данные
9. Запрос передается в приложение
10. Response возвращается пользователю
11. Logging и Reporting фиксируют события
```

## 📦 Независимые модули

Некоторые модули могут использоваться отдельно, если проекту не нужен весь фреймворк.

### Только Headers

```typescript
import { HeadersModule } from '@fab-orbita/shield/headers'

const headers = new HeadersModule()

app.use(headers.middleware())
```

---

### Только AI

```typescript
import { AIModule } from '@fab-orbita/shield/ai'

const ai = new AIModule()

app.use(ai.middleware())
```

---

### Только CSP

```typescript
import { CSPModule } from '@fab-orbita/shield/csp'

const csp = new CSPModule()

app.use(csp.middleware())
```

> Доступность отдельных subpath imports зависит от структуры пакета и экспортов в `package.json`.

## 🧪 Тестирование модулей

### Unit-тест модуля

```typescript
import { HeadersModule } from '@fab-orbita/shield/headers'

describe('HeadersModule', () => {
  it('should set security headers', () => {
    const headers = new HeadersModule()

    const req = createMockRequest()
    const res = createMockResponse()

    headers.apply(req, res)

    expect(res.getHeader('X-Content-Type-Options')).toBe('nosniff')
  })
})
```

---

### Integration-тест

```typescript
import request from 'supertest'
import express from 'express'
import { FABShield } from '@fab-orbita/shield'

describe('FAB Shield integration', () => {
  it('should protect Express app', async () => {
    const app = express()
    const shield = new FABShield()

    app.use(shield.middleware())

    app.get('/', (req, res) => {
      res.json({ ok: true })
    })

    const response = await request(app).get('/')

    expect(response.headers['x-content-type-options']).toBe('nosniff')
  })
})
```

## ⚠️ Важное замечание

FAB Shield — это application-level security framework. Он не заменяет полностью:

- внешний WAF;
- DDoS-защиту;
- аудит кода;
- pentest;
- DevSecOps-процессы;
- защиту инфраструктуры;
- корректную работу с секретами;
- юридический аудит соответствия требованиям.

Лучший результат достигается, когда модули FAB Shield используются как часть комплексной системы безопасности.
