## 📋 Введение

**FAB Shield** — это современный security-фреймворк, построенный на принципах:

- модульности;
- гибкости;
- производительности;
- расширяемости;
- наблюдаемости;
- application-level security.

В этом документе описана общая архитектура FAB Shield, его слои, компоненты, потоки данных и подходы к масштабированию.

## 🧩 Основные компоненты

## 1. Application Layer

**Назначение:** интеграция FAB Shield с веб-фреймворками.

### Компоненты

- Express middleware
- Fastify adapter
- Koa adapter
- NestJS integration
- Другие adapters

### Особенности

- единый подход к подключению;
- поддержка популярных Node.js-фреймворков;
- возможность добавления новых adapters;
- изоляция framework-specific логики;
- единый security-layer поверх разных фреймворков.

---

## 2. Middleware Layer

**Назначение:** основная логика защиты приложения.

### Компоненты

| Компонент | Назначение |
|:---|:---|
| **Headers Module** | Security-заголовки |
| **CSP Module** | Content Security Policy |
| **AI Module** | AI-анализ подозрительных запросов |
| **Plugins Module** | Расширение функциональности |
| **Rate Limiting Module** | Ограничение частоты запросов |
| **Rules Module** | Кастомные правила безопасности |
| **Security Module** | Валидация, sanitize, hardening |
| **Monitor Module** | Health check и контроль состояния |

### Особенности

- модульная структура;
- независимое включение и отключение модулей;
- гибкая настройка;
- возможность расширения через плагины;
- единый pipeline обработки запроса.

---

## 3. Core Layer

**Назначение:** базовые функции фреймворка и координация модулей.

### Компоненты

| Компонент | Назначение |
|:---|:---|
| **Config Manager** | Управление конфигурацией |
| **Request Handler** | Обработка входящих запросов |
| **Response Handler** | Обработка ответов |
| **Context Manager** | Управление request context |
| **Lifecycle Manager** | Инициализация и завершение работы |
| **Error Handler** | Централизованная обработка ошибок |

### Особенности

- единая точка входа;
- управление жизненным циклом;
- координация модулей;
- передача контекста между слоями;
- безопасная обработка ошибок.

---

## 4. Services Layer

**Назначение:** дополнительные сервисы, которые усиливают наблюдаемость и управляемость.

### Компоненты

| Компонент | Назначение |
|:---|:---|
| **Metrics Service** | Сбор метрик |
| **Reporting Service** | Генерация отчетов |
| **Logging Service** | Логирование |
| **Cache Service** | Кэширование |
| **Alert Service** | Уведомления |
| **Export Service** | Экспорт данных |

### Особенности

- сервисы могут быть независимыми;
- можно подключать только нужные части;
- архитектура рассчитана на расширение;
- удобна для мониторинга и аудита;
- поддерживает разные форматы экспорта.

---

## 5. Infrastructure Layer

**Назначение:** работа с внешними зависимостями и инфраструктурой.

### Компоненты

| Компонент | Назначение |
|:---|:---|
| **Database** | Хранение данных и истории событий |
| **Redis** | Кэширование и shared state |
| **File System** | Файловые логи и отчеты |
| **External APIs** | Интеграции с внешними сервисами |
| **Message Queue** | Асинхронные задачи и события |
| **Monitoring Stack** | Prometheus, Grafana, logs |

### Особенности

- абстракция от внешних систем;
- возможность замены реализаций;
- поддержка разных провайдеров;
- удобство масштабирования;
- подготовка к enterprise-интеграциям.

## 🔄 Потоки данных

## 1. Обработка запроса

```text
Request
   ↓
Application Layer
   ↓
Core Layer
   ↓
Context Manager
   ↓
Headers Module
   ↓
CSP Module
   ↓
AI Module
   ↓
Plugins Module
   ↓
Rate Limiting Module
   ↓
Rules Module
   ↓
Application Handler
   ↓
Response Handler
   ↓
Client Response
```

### Описание

1. Клиент отправляет запрос.
2. Запрос попадает в middleware FAB Shield.
3. Core создает request context.
4. Headers Module подготавливает security-заголовки.
5. CSP Module генерирует Content Security Policy.
6. AI Module анализирует запрос.
7. Plugins Module выполняет подключенные плагины.
8. Rate Limiting проверяет лимиты.
9. Rules Module применяет кастомные правила.
10. Запрос передается в приложение.
11. Ответ проходит response handler.
12. Клиент получает ответ.

---

## 2. Обработка угрозы

```text
AI Module
   ↓
Threat Detection
   ↓
Core Layer
   ↓
Rules Module
   ↓
Action Resolver
   ↓
Block / Challenge / Warn / Allow
   ↓
Metrics Service
   ↓
Logging Service
   ↓
Reporting Service
   ↓
Alert Service
```

### Возможные действия

| Действие | Описание |
|:---|:---|
| `allow` | Разрешить запрос |
| `warn` | Пропустить, но записать предупреждение |
| `challenge` | Запросить дополнительную проверку |
| `block` | Заблокировать запрос |
| `rate-limit` | Ограничить частоту |
| `log-only` | Только записать событие |

## 🎯 Принципы архитектуры

## 1. Модульность

Каждый компонент — независимый модуль.

```typescript
import { HeadersModule } from '@fab-orbita/shield/headers'
import { CSPModule } from '@fab-orbita/shield/csp'

const headers = new HeadersModule()
const csp = new CSPModule()

app.use(headers.middleware())
app.use(csp.middleware())
```

> Доступность отдельных subpath imports зависит от структуры пакета и экспортов в `package.json`.

---

## 2. Расширяемость

Новые возможности можно добавлять через плагины.

```typescript
import { FABShield } from '@fab-orbita/shield'

const shield = new FABShield({
  plugins: [
    {
      name: 'custom-plugin',

      middleware: (req, res, next) => {
        // Ваша логика
        next()
      }
    }
  ]
})
```

---

## 3. Гибкость

Можно включать только нужные функции.

```typescript
import { FABShield } from '@fab-orbita/shield'

const shield = new FABShield({
  headers: {
    enabled: true
  },

  csp: {
    enabled: false
  },

  ai: {
    enabled: true
  }
})
```

---

## 4. Производительность

Архитектура FAB Shield проектируется так, чтобы минимизировать overhead.

### Подходы

- lazy loading модулей;
- включение только нужных функций;
- кэширование результатов;
- оптимизация middleware pipeline;
- возможность отключать тяжелые проверки;
- экспорт метрик для контроля latency.

> Конкретные показатели производительности зависят от конфигурации, включенных модулей, плагинов и нагрузки приложения.

---

## 5. Наблюдаемость

FAB Shield проектируется с учетом observability.

### Что можно наблюдать

- количество запросов;
- заблокированные угрозы;
- CSP violations;
- rate limit events;
- response time;
- ошибки middleware;
- активность плагинов;
- состояние модулей.

## 🧪 Тестируемость архитектуры

Архитектура FAB Shield должна позволять тестировать модули независимо.

### Unit-тест

```typescript
import { HeadersModule } from '@fab-orbita/shield/headers'

describe('HeadersModule', () => {
  it('sets X-Content-Type-Options header', () => {
    const headers = new HeadersModule()
    const req = createMockRequest()
    const res = createMockResponse()

    headers.apply(req, res)

    expect(res.getHeader('X-Content-Type-Options')).toBe('nosniff')
  })
})
```

---

### Integration-тест

```typescript
import express from 'express'
import request from 'supertest'
import { FABShield } from '@fab-orbita/shield'

describe('FAB Shield middleware', () => {
  it('protects Express response', async () => {
    const app = express()
    const shield = new FABShield()

    app.use(shield.middleware())

    app.get('/', (req, res) => {
      res.json({ ok: true })
    })

    const response = await request(app).get('/')

    expect(response.headers['x-content-type-options']).toBe('nosniff')
  })
})
```

## 📦 Пример подключения

```typescript
import express from 'express'
import { FABShield } from '@fab-orbita/shield'

const app = express()

const shield = new FABShield({
  env: 'production',

  headers: {
    enabled: true
  },

  csp: {
    enabled: true,
    dynamic: true
  },

  ai: {
    enabled: true
  },

  monitoring: {
    enabled: true
  }
})

app.use(shield.middleware())

app.get('/', (req, res) => {
  res.json({
    message: 'Protected by FAB Shield'
  })
})

app.listen(3000, () => {
  console.log('🚀 Server started on http://localhost:3000')
})
```
