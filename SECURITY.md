## 📌 Введение

Безопасность является главным приоритетом проекта **FAB Shield**.

Мы серьёзно относимся к защите пользователей, приложений и инфраструктуры. Если вы нашли уязвимость, пожалуйста, сообщите о ней ответственно и не публикуйте детали до выпуска исправления.

Этот документ описывает:

- как сообщать об уязвимостях;
- какие версии поддерживаются;
- как мы классифицируем критичность проблем;
- как проходит процесс исправления;
- какие стандарты безопасности применяются в проекте;
- как работает программа признания исследователей безопасности.

---

## 🚨 Сообщение об уязвимости

### Важно

Не создавайте публичные GitHub Issues для уязвимостей.

Публичное раскрытие до исправления может поставить пользователей проекта под угрозу.

---

### Как сообщить

Отправьте письмо на email:

```text
derector@devorbit.ru
```

Рекомендуемая тема письма:

```text
[SECURITY] FAB Shield Vulnerability
```

В письме желательно указать:

- подробное описание уязвимости;
- версию FAB Shield;
- затронутый модуль или компонент;
- шаги для воспроизведения;
- пример конфигурации, если он важен;
- возможное влияние уязвимости;
- ваши контактные данные, если хотите получить обратную связь и благодарность.

## 🔄 Что происходит после сообщения

После получения сообщения об уязвимости мы действуем по следующему процессу:

| Этап | Срок | Что происходит |
|---|---:|---|
| Подтверждение | До 24 часов | Мы подтверждаем получение сообщения |
| Первичный анализ | 1–5 дней | Проверяем воспроизводимость и влияние |
| Классификация | 1–3 дня | Определяем уровень критичности |
| Исправление | До 30 дней | Готовим патч и тесты |
| Релиз | После проверки | Выпускаем исправленную версию |
| Раскрытие | После релиза | Публикуем информацию без опасных деталей |

Для критических уязвимостей процесс может быть ускорен.

## 🛡️ Стандарты безопасности проекта

FAB Shield разрабатывается как security-focused middleware для Node.js-приложений.

Основные направления защиты:

- security headers;
- Content Security Policy;
- rate limiting;
- обнаружение подозрительных payload-ов;
- защита от распространённых web-атак;
- плагины для расширения security-пайплайна;
- метрики и мониторинг security-событий.

---

## 🔐 Защита данных и конфигурации

Рекомендуемые стандарты для проектов, использующих FAB Shield:

- использовать HTTPS в production;
- хранить секреты только в переменных окружения;
- не коммитить `.env`, токены, ключи и пароли;
- использовать безопасные cookie-настройки;
- включать HSTS;
- включать CSP;
- регулярно обновлять зависимости;
- использовать dependency scanning;
- проверять security-логи и подозрительные события.

---

## 🧱 Защита от атак

| Тип атаки | Механизмы защиты |
|---|---|
| XSS | CSP, security headers, анализ подозрительных payload-ов |
| SQL Injection | Детекция SQL-паттернов, рекомендации по параметризованным запросам |
| NoSQL Injection | Детекция подозрительных операторов и payload-ов |
| Command Injection | Детекция опасных командных паттернов |
| Path Traversal | Детекция `../`, encoded traversal и похожих паттернов |
| Clickjacking | `X-Frame-Options`, CSP `frame-ancestors` |
| MIME Sniffing | `X-Content-Type-Options` |
| Brute Force | Rate limiting |
| DDoS-like bursts | Rate limiting и лимиты запросов |
| Information Leakage | Контроль заголовков и hardening response-ов |

FAB Shield повышает уровень защиты приложения, но не заменяет полноценный аудит безопасности, защищённую архитектуру, WAF, DevSecOps-процессы и безопасную работу с базами данных.

## 🧪 Проверка безопасности

### Рекомендуемые автоматические проверки

| Инструмент | Назначение |
|---|---|
| `npm audit` | Проверка известных уязвимостей в npm-зависимостях |
| Snyk | Сканирование зависимостей и supply-chain рисков |
| ESLint security rules | Поиск потенциально опасных паттернов в коде |
| GitHub CodeQL | Статический анализ кода |
| Trivy | Сканирование контейнеров и образов |
| Dependabot | Автоматические security-обновления зависимостей |

Если эти инструменты подключены в CI/CD, рекомендуется запускать их на каждый Pull Request и перед релизом.

### Рекомендуемые ручные проверки

- проверка OWASP Top 10;
- аудит конфигурации CSP;
- проверка rate limiting на чувствительных endpoint-ах;
- тестирование login/register/password-reset сценариев;
- проверка логирования security-событий;
- анализ false positive и false negative срабатываний;
- периодический аудит зависимостей.

## 🚦 Критичность уязвимостей

| Уровень | Примеры | Целевой срок исправления |
|---|---|---:|
| 🔴 Critical | RCE, обход аутентификации, утечка секретов | 24–72 часа |
| 🟠 High | повышение привилегий, доступ к чувствительным данным | до 7 дней |
| 🟡 Medium | DoS, обход отдельных защитных правил | до 14 дней |
| 🟢 Low | minor hardening issues, документационные security-ошибки | до 30 дней |

Сроки могут зависеть от сложности уязвимости и необходимости обратной совместимости.

## 🚀 Процесс выпуска патча

1. Получение сообщения.
2. Подтверждение получения.
3. Воспроизведение проблемы.
4. Оценка влияния.
5. Разработка исправления.
6. Добавление regression-тестов.
7. Проверка сборки и тестов.
8. Выпуск patch/minor-релиза.
9. Обновление `CHANGELOG.md`.
10. Публикация security advisory, если требуется.

## 📝 Ответственное раскрытие

Мы приветствуем responsible disclosure.

### Просим исследователей

- сообщать об уязвимости до публичной публикации;
- дать разумное время на исправление;
- не использовать уязвимость для доступа к чужим данным;
- не нарушать работу чужих сервисов;
- не публиковать exploit-код до выхода патча;
- предоставить понятные шаги воспроизведения.

### Мы обещаем

- подтвердить получение сообщения;
- рассмотреть отчёт добросовестно;
- не игнорировать security-issues;
- сообщить о результате проверки;
- выпустить исправление при подтверждении проблемы;
- указать исследователя в благодарностях, если он этого хочет.

## 🔄 Обновления безопасности

Рекомендуется использовать актуальную версию пакета:

```bash
npm install @fab-orbita/shield@latest
```

Для проектов, где допустимы совместимые обновления в пределах major-версии:

```json
{
  "dependencies": {
    "@fab-orbita/shield": "^1.3.0"
  }
}
```

Рекомендации:

- следите за `CHANGELOG.md`;
- проверяйте GitHub Releases;
- обновляйте зависимости регулярно;
- включите Dependabot или аналогичный инструмент;
- тестируйте обновления в staging перед production.

## ✅ Чек-лист безопасности для пользователей

- [ ] Используется HTTPS.

- [ ] Используется dependency scanning.

## 🚫 Что FAB Shield не заменяет

FAB Shield помогает усилить безопасность Node.js-приложений, но не заменяет:

- полноценный внешний WAF;
- аудит исходного кода;
- penetration testing;
- secure SDLC;
- DevSecOps-процессы;
- защиту инфраструктуры;
- безопасную архитектуру приложения;
- корректную бизнес-валидацию;
- безопасную работу с базой данных;
- управление секретами на уровне инфраструктуры.
